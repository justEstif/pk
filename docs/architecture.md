# pk Architecture

## Two-store model

pk separates project knowledge into two stores with distinct roles.

| Store        | Technology            | Question it answers | Lifecycle                                                |
| ------------ | --------------------- | ------------------- | -------------------------------------------------------- |
| **Semantic** | SQLite (`.index.db`)  | What do we know?    | Written on `pk new`/`pk delete`, indexed on `pk rebuild` |
| **Episodic** | Git (commits + notes) | What happened?      | Written automatically on every pk command                |

### Semantic store

Plain markdown files with structured frontmatter, indexed into SQLite FTS5 for full-text search.

**Note types:** `source`, `note`, `decision`, `question`, `index` — each with required sections and valid status transitions (see `skill/references/knowledge-model.md`).

**Retrieval:** FTS5 with porter stemming (current). Hybrid FTS + vector search planned — same `.index.db`, new `note_vectors` table, fused BM25/cosine ranking. No new commands; `pk search` upgrades transparently when embeddings are configured.

### Episodic store

Git as the event log. Two kinds of events:

- **Commits** — mutations: `knowledge: intake note X`, `knowledge: delete decision Y`
- **Git notes on HEAD** — observations: `pk prime`, `pk search`, `pk synthesize`

Commits keep `git log` clean (only file changes). Notes capture read-only operations without polluting the commit graph. `pk history` interleaves both into a unified timeline.

Sessions emerge from time gaps in the event stream — no explicit start/end markers, no agent cooperation required.

## Synthesis layer

`pk synthesize` assembles context from both stores for agent consumption. It is retrieval + formatting, not generation. No LLM calls.

Current: selects and ranks notes from the semantic store only.
Target: selects notes + recent episodic events, combined into a single context block.

Output is always ephemeral (stdout). The agent decides what to persist via `pk new`.

## Embedding seam

Config: `pk config --embedding <provider/model>` (field already exists).
Provider interface:

```
EmbeddingProvider { embed(text) → float[], dimensions() → int }
```

Implementations: `OllamaProvider` (local, offline-first), `OpenAIProvider` (cloud, zero setup). Empty config = embeddings disabled, FTS-only.

`pk rebuild` gains an embedding pass: read notes → FTS insert → embed title+body → store vector. Batch operation, no incremental embedding at `pk new` time.

`pk search` runs hybrid when vectors exist: FTS BM25 + cosine similarity, fused with weighted average. Fallback to FTS-only when unconfigured.

## Harness integration

Two harnesses: OpenCode (plugin), Pi (extension). All interact via `pk` CLI with `--json`. `pk prime` is the injection source. Each harness must support shell-level env injection so `PK_KNOWLEDGE_DIR` is available to every tool call.

Harnesses do not need to know about the episodic store — git events are generated by pk commands the agent already calls, not by agent lifecycle hooks.
